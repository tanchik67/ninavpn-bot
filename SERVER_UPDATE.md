# Как обновить бота на сервере (простая инструкция)

Эта инструкция для случая, когда **новая версия кода уже лежит на вашем Mac** (папка `ninavpn-bot`), и её нужно **скопировать на сервер** и **перезапустить бота**.

Первый раз ставить бота с нуля — смотрите **[DEPLOY.md](DEPLOY.md)**.

---

## Ваш сервер

| Параметр | Значение |
|----------|----------|
| **Адрес (IP)** | `2.27.123.28` |
| **Папка бота на сервере** | `/opt/ninavpn-bot` |

Если когда‑нибудь сменится IP или домен — во всех командах ниже замените **`2.27.123.28`** на новый адрес.

---

## Что понадобится

| Что | Зачем |
|-----|--------|
| Компьютер Mac | Откуда копируем файлы |
| Доступ к серверу | Логин и пароль **или** ключ (как вам выдавали при настройке) |
| Логин на сервере | В командах ниже по умолчанию указан **`root`**. Если вы заходите под другим именем — замените слово **`root`** перед символом **`@`** на своё имя пользователя. |

**Важно:** на сервере уже должна быть папка с ботом (`/opt/ninavpn-bot`). Если бот там раньше работал — всё в порядке.

## Шаг 1. Открыть «Терминал» на Mac

1. Нажмите **Spotlight** (лупа вверху справа) или **Cmd + Пробел**.
2. Введите **Терминал** и откройте программу **Terminal**.

Дальше все команды вводятся **в это окно** и выполняются клавишей **Enter**.

---

## Шаг 2. Скопировать всю папку бота на сервер (обычный способ обновления)

Смысл: **скопировать папку `ninavpn-bot` с Mac на сервер** `2.27.123.28`, не трогая на сервере пароли (файл `.env`) и базу данных.

### 2.1. Где лежит проект на Mac

По умолчанию ниже считается, что папка **`ninavpn-bot`** лежит в **Загрузках** (`Downloads`). Если у вас другой путь — в команде `cd` перейдите в папку **на уровень выше** вашей папки `ninavpn-bot` (чтобы рядом оказалась именно папка `ninavpn-bot`).

### 2.2. Команда копирования (готовая строка)

Сначала:

```bash
cd ~/Downloads
```

Потом выполните **одну длинную строку** (логин по умолчанию **`root`**; если не подходит — замените только `root`):

```bash
rsync -avz --exclude 'venv' --exclude '__pycache__' --exclude '*.db' --exclude '.env' --exclude '.git' --exclude '.DS_Store' --exclude '._*' ninavpn-bot/ root@2.27.123.28:/opt/ninavpn-bot/
```

- Если спросят **пароль** — введите пароль от сервера (символы могут не отображаться — это нормально).
- Если настроен **ключ** — пароль может не спросить.

**Почему в команде столько слов с `exclude`:** мы **не** перезаписываем на сервере файл с паролями (`.env`), базу (`*.db`) и служебные папки — на сервере они должны остаться своими.

---

## Шаг 2 (альтернатива). Отправить один файл на сервер

Иногда нужно залить **один файл** (например исправленный скрипт), а не всю папку.

1. Узнайте **полный путь к файлу** на Mac (в Finder: правый клик по файлу → зажать **Option** → **Скопировать путь к …**).
2. В Терминале выполните (подставьте **свой путь** вместо примера):

```bash
scp "/полный/путь/к/файлу.py" root@2.27.123.28:/opt/ninavpn-bot/
```

Файл попадёт **в корень** папки бота на сервере. Если нужна подпапка (например `handlers`), укажите её в конце:

```bash
scp "/полный/путь/к/файлу.py" root@2.27.123.28:/opt/ninavpn-bot/handlers/
```

**Не копируйте** поверх серверный файл **`.env`** с Mac, если на Mac лежит тестовая копия — на сервере должны остаться боевые пароли и токены.

После замены одного файла всё равно зайдите на сервер и перезапустите бота (шаг 3).

---

## Шаг 3. Зайти на сервер и перезапустить бота

После копирования нужно **обновить зависимости** (если менялись библиотеки) и **перезапустить службу**.

### 3.1. Подключение к серверу

В Терминале на Mac:

```bash
ssh root@2.27.123.28
```

Если логин не `root`:

```bash
ssh ВАШ_ЛОГИН@2.27.123.28
```

Вы окажетесь «внутри» сервера (приглашение вроде `root@...`).

### 3.2. Запуск скрипта обновления

Сначала **обязательно перейдите в каталог с ботом** — иначе в домашней папке (`root@...:~#`) команда `bash scripts/update-server.sh` выдаст **«No such file or directory»**: там просто нет папки `scripts`.

По очереди:

```bash
cd /opt/ninavpn-bot
bash scripts/update-server.sh
```

Если каталог другой (бот ставили не в `/opt/ninavpn-bot`), подставьте свой путь. Проверить, где лежит проект, можно так:

```bash
ls /opt/ninavpn-bot/scripts/update-server.sh
```

Если файла нет — поищите на сервере: `find /opt /home -name update-server.sh 2>/dev/null`

Скрипт сам установит нужные библиотеки и перезапустит бота.

### 3.3. Проверка

В Telegram напишите боту **`/start`**. Если ответ есть — обычно всё прошло успешно.

## Если что-то пошло не так

| Симптом | Что попробовать |
|--------|------------------|
| «Команда не найдена: rsync» | На Mac `rsync` обычно уже есть. Обновите macOS или установите **Xcode Command Line Tools** (в интернете: «install command line tools mac»). |
| «Permission denied» при SSH | Неверный логин/пароль или не тот ключ. Уточните у того, кто настраивал сервер. |
| `bash: scripts/update-server.sh: No such file or directory` | Вы не в каталоге бота. Сначала **`cd /opt/ninavpn-bot`** (или туда, куда установлен проект), потом снова **`bash scripts/update-server.sh`**. |
| Бот молчит после обновления | На сервере выполните: `sudo journalctl -u ninavpn-bot -n 50 --no-pager` и посмотрите строки с ошибкой. |
| В логах **`TelegramUnauthorizedError` / `Unauthorized`** | Токен в **`/opt/ninavpn-bot/.env`** не совпадает с **@BotFather** (отозван, опечатка, пробелы). Проверка: `curl -s "https://api.telegram.org/bot<токен>/getMe"` → `"ok":true`. После правки — **`sudo systemctl restart ninavpn-bot`**. В актуальном коде **`BOT_TOKEN` из `.env` важнее**, чем переменная окружения `BOT_TOKEN` (systemd/export), чтобы не подставлялся старый токен. Проверьте: **`grep BOT_TOKEN /etc/environment`** и **`systemctl show ninavpn-bot -p Environment`** — лишний `BOT_TOKEN` лучше убрать. |
| `status=200/CHDIR`, сервис в цикле перезапуска | В **`/etc/systemd/system/ninavpn-bot.service`** поле **`WorkingDirectory`** должно указывать на каталог, где лежат **`main.py`** и **`.env`**. Частая ошибка: путь **`/opt/ninavpn-bot/ninavpn-bot`**, хотя проект лежит сразу в **`/opt/ninavpn-bot`**. Исправьте на `WorkingDirectory=/opt/ninavpn-bot`, затем **`sudo systemctl daemon-reload`** и **`sudo systemctl restart ninavpn-bot`**. |
| После копирования с Mac бот «ломается» с ошибкой про Python | Возможно, случайно скопировалась папка `venv` с Mac. На сервере: `cd /opt/ninavpn-bot` и `bash scripts/ninavpn-heal.sh --recreate-venv`. |

## Краткий чеклист

1. На Mac: `cd ~/Downloads` → команда **`rsync`** с **`root@2.27.123.28`** (или своим логином).
2. На сервере: **`ssh root@2.27.123.28`** → **`cd /opt/ninavpn-bot`** → **`bash scripts/update-server.sh`**.
3. В Telegram: **`/start`** у бота.

Если нужно поменять настройки (токены, ключи) — правьте на **сервере** файл **`/opt/ninavpn-bot/.env`**, затем снова **`sudo systemctl restart ninavpn-bot`**.

## Обновление данных на сервере

Здесь «данные» — это **не исходный код**, а то, что хранится на сервере рядом с ботом: база, кэш, настройки.

### База данных

- Файл **`*.db`** (часто `ninavpn.db`) **не копируется** с Mac командой `rsync` выше — на сервере остаётся **рабочая** база пользователей и платежей.
- Если вы **переезжаете** на новый сервер или **перед крупным изменением** — сделайте копию базы с сервера (скачать в безопасное место). Путь к файлу смотрите в **`DATABASE_URL`** в `.env` на сервере.
- При **PostgreSQL** и смене схемы после обновления кода может понадобиться миграция: см. **[DEPLOY.md](DEPLOY.md)** (раздел про Alembic). При **SQLite** новые таблицы создаются при старте бота, если это предусмотрено версией кода.

### Файл настроек `.env`

- Все **секреты** (токен бота, ключи панелей VPN, `PAYMENT_PUBLIC_BASE_URL` и т.д.) правятся **только на сервере** в **`/opt/ninavpn-bot/.env`**.
- После любого изменения `.env`: **`sudo systemctl restart ninavpn-bot`**.

### Кэш Geo для v2ray (`/geo/geoip.dat`, `/geo/geosite.dat`)

Бот может кэшировать большие файлы **geoip.dat** / **geosite.dat** в папке **`data/v2ray_geo/`** на сервере (см. [DEPLOY.md](DEPLOY.md), HTTP-сервис).

- Обычно **ничего делать не нужно**: файлы сами обновляются по истечении времени кэша (по умолчанию несколько часов) или при следующем запросе пользователя.
- Чтобы **принудительно скачать свежие файлы** с GitHub при следующем обращении к `/geo/...`:

```bash
cd /opt/ninavpn-bot
rm -f data/v2ray_geo/geoip.dat data/v2ray_geo/geosite.dat
sudo systemctl restart ninavpn-bot
```

(Папку `data/v2ray_geo/` удалять не обязательно — достаточно удалить **`.dat`** файлы.)

### Nginx и публичные ссылки

Если пользователи открывают Geo по адресу **`https://ваш-домен/geo/...`**, убедитесь, что в nginx есть **`location /geo/`** → прокси на порт бота (**8080**), как в **[DEPLOY.md](DEPLOY.md)**. В **`.env`** должны быть **`PAYMENT_PUBLIC_BASE_URL`** или **`PUBLIC_WEB_BASE_URL`**, чтобы в боте подставлялись правильные ссылки в разделе «Как подключиться».

## Несколько копий бота на сервере: какая «рабочая» и что удалить

Если на сервере копировали проект в разные папки, важно **не трогать ту копию**, из которой реально стартует **systemd**, и **не** держать второй процесс с тем же **BOT_TOKEN** (будет конфликт polling).

### 1. Узнать, какой каталог использует служба

Выполните на сервере:

```bash
systemctl cat ninavpn-bot.service
systemctl show ninavpn-bot -p FragmentPath -p ExecMainStart -p WorkingDirectory --no-pager
```

Запомните значения **`WorkingDirectory=`** и **`ExecStart=`** (путь к `python` и к **`main.py`**). Обычно это **`/opt/ninavpn-bot`** — **её и считаем основной рабочей копией.**

Проверка, что процесс действительно из этой папки:

```bash
PID=$(systemctl show ninavpn-bot -p MainPID --value)
[ -n "$PID" ] && [ "$PID" != "0" ] && readlink -f "/proc/$PID/cwd"
ls -la "/proc/$PID/exe" 2>/dev/null
```

Текущий каталог процесса (`cwd`) должен совпадать с **`WorkingDirectory`**.

### 2. Найти другие копии

```bash
sudo find /opt /home /root /var/www -maxdepth 6 -type f -path '*/ninavpn-bot/main.py' 2>/dev/null
sudo find /opt /home /root -maxdepth 5 -type d -name 'ninavpn-bot' 2>/dev/null
```

Также проверьте **cron** и **screen/tmux** (не запущен ли второй бот вручную):

```bash
crontab -l 2>/dev/null; sudo crontab -l 2>/dev/null
```

### 3. Что удалять (осторожно)

- **Не удаляйте** каталог, который указан в **`WorkingDirectory`** и **`ExecStart`** у **`ninavpn-bot.service`**, пока не перенесёте службу на другой путь и не проверите запуск.
- Перед удалением любой **другой** копии убедитесь, что из неё **ничего не запущено** (нет второго `python main.py` с тем же ботом):

```bash
ps aux | grep -E '[p]ython.*main\.py'
```

Если видите процесс из «лишней» папки — остановите его (завершите PID или закройте сессию), затем можно удалить **только эту** папку, например:

```bash
# Пример: удалить дубликат НЕ из рабочего /opt/ninavpn-bot (подставьте свой лишний путь!)
# sudo rm -rf /путь/к/старой/копии/ninavpn-bot
```

Надёжнее сначала переименовать (на случай ошибки):

```bash
# sudo mv /путь/к/дубликату/ninavpn-bot /путь/к/дубликату/ninavpn-bot.bak-removed
```

### 4. После чистки

```bash
sudo systemctl restart ninavpn-bot
sudo systemctl status ninavpn-bot
```

В Telegram — **`/start`**. В логах не должно быть повторяющихся конфликтов и **Unauthorized**, если **`.env`** с **BOT_TOKEN** только в **одной** рабочей копии.

### 5. Один BOT_TOKEN — один процесс

Два запущенных бота с **одним и тем же токеном** вызывают нестабильность. Держите **одну** установку и один **`systemctl restart ninavpn-bot`**.

### 6. Ошибка в логах: `no such column: payments.admin_notify_sent`

Схема SQLite **устарела** после обновления кода — нужно применить миграции Alembic:

```bash
cd /opt/ninavpn-bot
./venv/bin/python scripts/apply_migrations.py
sudo systemctl restart ninavpn-bot
```

Если скрипта ещё нет на сервере — скопируйте **`scripts/apply_migrations.py`** с Mac или выполните вручную (из каталога бота, с активированным `venv` и с тем же **`DATABASE_URL`**, что в **`.env`**):

```bash
source venv/bin/activate
set -a && . ./.env && set +a
alembic upgrade head
```

### 7. `alembic upgrade`: `ValueError: source code string cannot contain null bytes`

Чаще всего в **`alembic/versions/`** лежат мусорные файлы с Mac — **`._*.py`** (AppleDouble). Alembic пытается открыть их как миграции и падает.

На сервере удалите такие файлы по всему проекту (не трогая обычные `.py`):

```bash
cd /opt/ninavpn-bot
find alembic/versions -name '._*' -ls
find alembic/versions -name '._*' -delete
find . -maxdepth 3 -name '._*' -delete
```

Проверьте, что в **`alembic/versions/`** остались только **`001_initial_schema.py`** и **`002_payment_admin_notify.py`** (имена могут слегка отличаться, главное — без префикса **`._`**). Затем снова примените миграции **только из каталога бота** и **с тем же `DATABASE_URL`, что в `.env`** (удобнее всего скриптом выше — он сам подхватывает `.env`).

Чтобы при следующем **rsync** с Mac не копировать `._*`, добавьте в команду: **`--exclude '.DS_Store' --exclude '._*'`**.

### 8. «Пропало всё»: другие тарифы, нет Т‑Банка, пустые пользователи

**Миграции Alembic не стирают таблицы и не меняют `.env`.** Такие симптомы обычно значат одно из двух.

**А) Бот смотрит на другую базу SQLite**, чем раньше (или на новый пустой файл). Для относительного пути `sqlite+aiosqlite:///ninavpn.db` файл всегда создаётся/открывается **относительно рабочего каталога процесса** — у systemd это **`WorkingDirectory`** из юнита (ожидается **`/opt/ninavpn-bot`**). Если миграции запускали **не из** `/opt/ninavpn-bot` и вручную задавали `export DATABASE_URL=...`, Alembic мог обновить **другой** `ninavpn.db` (например в `/root`), а сервис по-прежнему читает **`/opt/ninavpn-bot/ninavpn.db`** — или наоборот.

Что сделать на сервере:

```bash
cd /opt/ninavpn-bot
grep -E '^DATABASE_URL=' .env
ls -la ninavpn.db *.db 2>/dev/null
```

Ищите **резервные копии** базы (`ninavpn.db.bak`, старый снимок, бэкап хостинга). Если найдёте файл с «настоящими» данными — остановите бота, **замените** им текущий `ninavpn.db` (или поправьте **`DATABASE_URL`** в `.env` на абсолютный путь к тому файлу), снова **`./venv/bin/python scripts/apply_migrations.py`**, затем **`sudo systemctl restart ninavpn-bot`**.

**Б) Пропал только Т‑Банк** (крипта и перевод по ссылке остались) — в **эффективном** окружении процесса нет обоих **`TBANK_TERMINAL_KEY`** и **`TBANK_PASSWORD`**. Кнопка появляется только если в **`.env`** заданы **оба** значения (без пробелов/кавычек по краям). Проверьте: **`grep TBANK /opt/ninavpn-bot/.env`**. Убедитесь, что **`EnvironmentFile=`** в **`ninavpn-bot.service`** указывает на тот же **`.env`**, который вы правите, и нет дублирующего пустого `TBANK_*` в **`/etc/environment`** или в **`Environment=`** юнита.

**Тарифы:** если в таблице **`plan_tariffs`** есть хотя бы одна активная строка, витрина берётся **из БД**. Если база пустая или вы смотрите на новый файл — витрина станет **встроенной** из кода (`config.PLANS` — другие цены/названия). Восстановление — снова **правильный файл БД** или ручное заполнение **`plan_tariffs`** в SQLite.
